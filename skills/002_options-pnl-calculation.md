# Options Premium & PnL Calculation Standard

本文件定義了期權交易中「權利金 (Premium)」的取得來源，以及「未實現損益 (Unrealized PnL)」的精確比例分配計算邏輯。當未來需要維護或擴充 IB 報表解析 (例如 `import-ib/route.ts`) 與 PnL 呈現時，必須嚴格遵守此演算法。

## 1. 權利金 (Premium) 的精確取得

在解析 Interactive Brokers (IB) 報表的「交易 (Trades)」區塊時，必須注意各欄位的財務意義：
- **收益 (Proceeds)**：僅為 `交易價格 × 口數 × 合約乘數`，**不包含**交易成本。
- **佣金/稅 (Comm/Tax)**：交易產生的手續費支出。
- **基礎 (Basis)**：等於 `收益 + 佣金/稅`，代表該筆交易實際產生的淨現金流。

**【強制實作規範】**
存入資料庫的 `premium` 必須讀取 **「基礎 (Basis)」** 欄位的絕對值，而非「收益」。
因為身為賣方 (Sell Open)，權利金是收入，但同時也是帳上的成本基礎 (Cost Basis)。使用「基礎」才能將手續費完美攤提進期權的開倉成本中。
```typescript
// 讀取「基礎」欄位 (通常位於活動賬單的 off+7 或 off+8)
const basisStr = cols[off + 7]; 
// 只針對 Sell Open 紀錄權利金 (Buy Open 的 premium 記為 0)
const premium = quantity > 0 ? 0 : Math.abs(parseNumber(basisStr)); 
```

## 2. 未實現損益 (Unrealized PnL) 的比例分配演算法

IB 報表的「未平倉持倉 (Open Positions)」區塊會將相同標的、履約價、到期日的合約**合併為一行**，並給出「總數量」、「總價值」與「總未實現損益」。

**【常見錯誤警告】**
❌ **絕對不可**直接把報表上的「總未實現損益」寫入個別的每一筆歷史交易中。
❌ **絕對不可**直接按口數比例平攤「總未實現損益」。因為同一份合約的分批賣出紀錄，可能賣在完全不同的權利金價格。若直接平攤 PnL，會抹殺掉不同進場點的價格優勢。

**【正確演算法：價值分配法】**
我們必須按比例分配的是合約當下的**「總價值 (Value)」**，再讓單筆交易用這個分配到的價值，去扣除它自己專屬的**「初始權利金成本」**。

1. **取得總價值**：從「未平倉持倉」區塊解析出 `value` (總價值)。
2. **分配單筆價值 (Prorated Value)**：`單筆價值 = (總價值 / 總口數) * 單筆口數`
3. **計算單筆淨損益 (Trade PnL)**：`單筆淨損益 = 單筆價值 - 單筆成本基礎`
   - 對於賣方 (`opt.quantity < 0`)，因為收了權利金，成本基礎記為 `-opt.premium`。
   - 推導公式：`單筆淨損益 = 單筆價值 - (-opt.premium) = 單筆價值 + opt.premium`

```typescript
// 在更新現有合約 或 新增合約時的標準計算式
const proratedValue = (pos.value / pos.quantity) * opt.quantity;
// opt.quantity < 0 代表 Sell Open
const tradePnl = proratedValue + (opt.quantity < 0 ? opt.premium : -opt.premium);
```

## 3. 系統 MTM PnL 與 IB 報表的差異 (優越性)

在 IB 報表的「交易」區塊中，會有一欄「按市值計算的損益 (MTM PnL)」。
- **IB 的 MTM PnL**：它是用 `單筆價值 - 收益 (Proceeds)` 算出來的。**它完全忽略了開倉手續費**，因此算出來的利潤會虛高。
- **本系統的 PnL**：如上所述，我們使用包含手續費的「基礎 (Basis)」計算 `opt.premium`。因此我們算出的單筆損益是**真正扣除成本後的淨損益**，精確度超越 IB 原廠報表該欄位的顯示數字。
- 此外，IB 的 MTM PnL 只會顯示在當天有交易的明細中；而本系統的價值分配演算法，能讓**所有歷史庫存**在每天匯入報表時，都獲得最精準的更新。

## 4. 儀表板「權利金-季 / 權利金-年」的計算邏輯

在 `OptionsSummaryPanel` 等總覽儀表板中，所謂的「權利金-季」與「權利金-年」在定義上**並不是**單純加總收到的現金權利金，而是代表特定時間範圍內的**真實淨損益總和**。

**【計算邏輯與公式】**
1. **時間範圍基礎**：以 `open_date` (開倉日) 為基準，撈出在「本季」或「本年」內開倉的所有交易 (包含期權與相關股票)。
2. **加總指標**：加總這些交易的 `final_profit`。
   - **已平倉 (Closed)** 的部位：`final_profit` 為「已實現損益 (Realized PnL)」。
   - **未平倉 (Open)** 的部位：`final_profit` 為「未實現損益 (Unrealized PnL)」。
3. **商業意義**：
   此數字呈現的是**「在該季/該年內佈局的部位，截至目前為止共創造了多少真實利潤」**。
   當遇到大盤劇烈波動，導致帳上未平倉的部位有巨大的未實現虧損時，該欄位極可能會顯示為負數。這是系統為了反映「最真實的部位健康度」，而非美化過的現金流入帳面數字。
